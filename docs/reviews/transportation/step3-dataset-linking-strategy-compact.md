# Step 3 Dataset Linking Strategy Compact Review

- 기준 문서: `AGENTS.md`, `docs/plans/transportation/00_EXECUTION_ROADMAP.md`, `docs/plans/transportation/01_TRANSIT_DATA_SOURCE_SURVEY.md`, `docs/plans/transportation/03_DATASET_LINKING_STRATEGY.md`
- 작업 일자: 2026-04-15
- 작업 범위: `ODsay`, `부산 BIMS`, `부산교통공사/odcloud 지하철 운행정보` 간 연결 전략 정의

## 목적

Step 1, Step 2에서 확인한 대중교통 데이터 원천을 기준으로,
출발지-도착지 경로 후보는 `ODsay`,
버스 정류장/도착시간은 `부산 BIMS`,
지하철 역명/정거장 시각 정보는 `부산교통공사/odcloud`
로 역할을 분리했을 때 실제로 어떤 키와 규칙으로 연결할 수 있는지 정리한다.

이번 단계의 목표는 완전 자동 통합이 아니라, MVP에서 사용할 수 있는 최소 연결 전략과
수동 보정이 필요한 구간을 구분하는 것이다.

## 입력 정리

### 1. ODsay
- 출발지 / 도착지 기반 대중교통 경로 후보 조회 가능
- `searchPubTransPathT` 기준 `totalTime`, `busTransitCount`, `subwayTransitCount` 확인
- Step 2 스모크 테스트 기준 경로 후보 조회 성공
- 근거:
  - `scripts/transportation/run_transit_api_smoke_tests.py:73-116`
  - `docs/reviews/transportation/step2-transit-smoke-test-script-bootstrap-compact.md`

### 2. 부산 BIMS
- `busStopList`로 정류장명, ARS 번호, `bstopid`, 좌표 조회 가능
- `stopArrByBstopid`로 `lineid`, `lineno`, `min1`, `station1`, `lowplate1`, `min2`, `station2`, `lowplate2` 확인
- 버스 실시간 도착시간 원천으로 사용 가능
- 근거:
  - `scripts/transportation/run_transit_api_smoke_tests.py:119-181`
  - `docs/reviews/transportation/step1-transit-api-access-smoke-test-compact.md`

### 3. 부산교통공사/odcloud 지하철 운행정보
- 노선명, 노선번호, 운행구간 시작역/종점역, 운행구간정거장, 정거장도착시각, 정거장출발시각, 운행속도 확인
- 실시간 도착예정이 아니라 기준 운행 정보 또는 시간표성 정보로 해석하는 것이 적절
- 근거:
  - `scripts/transportation/run_transit_api_smoke_tests.py:184-216`
  - `docs/reviews/transportation/step1-transit-api-access-smoke-test-compact.md`

## 결론 요약

- 경로 후보와 총 이동시간의 기준 앵커는 `ODsay`로 둔다.
- 버스 구간의 정류장 식별과 실시간 도착시간은 `부산 BIMS`로 보강한다.
- 지하철 구간의 역명, 정거장 순서, 도착/출발 시각은 `부산교통공사/odcloud`로 보강한다.
- 데이터셋 간 ID 체계가 완전히 통일되어 있지 않으므로 `직접 ID 매핑`만으로는 부족하다.
- MVP에서는 `이름 정규화 + 좌표 근접 + 노선 보조키` 조합 전략이 현실적이다.
- 100% 자동 매핑을 목표로 두지 않고, 실패 케이스는 수동 보정 대상으로 분리한다.

## 데이터셋별 역할 분담

| 데이터셋 | 주 역할 | Step 3 연결 기준 |
| --- | --- | --- |
| ODsay | 출발지-도착지 경로 후보, 총 이동시간, 환승 수, 구간 타입 | 전체 응답의 앵커 |
| 부산 BIMS | 버스 정류장, 노선, 실시간 도착시간, 저상버스 여부 | 버스 구간 보강 |
| 부산교통공사/odcloud | 지하철 노선, 역명, 정거장 시각, 운행구간 문자열 | 지하철 구간 보강 |

## 연결 전략

### 1. 전체 앵커 전략

- 전체 경로 후보는 `ODsay` 응답을 기준 구조로 삼는다.
- 이유:
  - 출발지 / 도착지 기반 경로 후보 조회가 가능하다.
  - 총 이동시간과 구간 수를 바로 확보할 수 있다.
  - 버스 / 지하철 / 환승이 포함된 후보를 한 번에 가져올 수 있다.

즉, Step 3의 연결은
`ODsay segment -> 버스면 BIMS 보강 -> 지하철이면 odcloud 보강`
형태로 진행하는 것이 맞다.

### 2. 버스 구간 연결 전략

우선순위는 아래와 같다.

1. `노선 번호 + 정류장명` 일치
2. `노선 ID + 정류장명` 일치
3. `정류장명 정규화 + 좌표 근접` 일치
4. 실패 시 `후보 목록`으로 남기고 수동 보정

버스는 실시간 도착정보가 중요하므로, 최종적으로는 `BIMS bstopid`를 확보해야 한다.
따라서 버스 구간은 “경로 후보 조회”보다 “정류장 식별자 확보”가 핵심이다.

#### 버스 매핑에 사용할 키
- `ODsay`: 버스 구간 시작 정류장명, 종료 정류장명, 노선 번호 또는 노선 식별자
- `BIMS`: `bstopid`, `arsno`, `bstopnm`, `lineid`, `lineno`, 좌표

#### 버스 정규화 규칙
- 괄호, 공백, 점(`.`), 출구 표기, 방향 표기를 제거한 비교용 이름을 만든다.
- 예:
  - `서면역.서면지하상가` -> `서면역서면지하상가`
  - `서면역 2번출구` -> `서면역2번출구`
- 정규화 후 완전 일치가 아니면 좌표 근접 비교를 추가한다.

#### 버스 MVP 판단
- `ODsay 버스 구간`에서 탑승 정류장명과 하차 정류장명을 추출할 수 있다면
  `busStopList` 또는 정류장 캐시와 정규화 매칭을 먼저 시도한다.
- 한 번 `bstopid`를 확보하면 `stopArrByBstopid`로 도착시간을 붙일 수 있다.
- 저상버스 여부는 `lowplate1`, `lowplate2`를 그대로 보강 필드로 사용한다.

### 3. 지하철 구간 연결 전략

우선순위는 아래와 같다.

1. `노선명 + 역명` 정규화 일치
2. `노선번호 + 역명` 일치
3. `운행구간정거장` 문자열 안 포함 여부 확인
4. 실패 시 동일 노선 후보 다건을 남기고 수동 보정

지하철은 실시간 도착예정보다 “정거장 순서와 시간표성 도착/출발 시각”이 핵심이다.
따라서 `역명 단일 키`보다 `노선 문맥 안의 역명`으로 매칭하는 것이 더 안정적이다.

#### 지하철 매핑에 사용할 키
- `ODsay`: 지하철 노선명, 탑승역명, 하차역명, 구간 이동시간
- `odcloud`: 노선명, 노선번호, 운행구간 시작역/종점역, 운행구간정거장, 정거장도착시각, 정거장출발시각

#### 지하철 MVP 판단
- `ODsay 지하철 구간`의 노선명과 탑승/하차역명을 기준으로 `odcloud` 레코드를 좁힌다.
- 후보 레코드의 `운행구간정거장` 문자열 안에 두 역이 모두 포함되는지 본다.
- 포함된다면 해당 레코드의 정거장도착시각 / 정거장출발시각을 파싱해 보강한다.
- 단, 이 값은 “실시간 예정”이 아니라 “시간표성 정보”로 표시해야 한다.

## 직접 ID 매핑 가능성

### 판단
- Step 3 기준 직접 ID 매핑만으로 해결하는 전략은 부적절하다.

### 이유
- `ODsay`, `BIMS`, `부산교통공사`가 동일한 정류장/역 ID 체계를 공유한다는 근거가 없다.
- BIMS는 `bstopid`, `lineid` 중심이고,
  지하철 공공데이터는 노선명/역명/운행구간 문자열 중심이다.
- 따라서 데이터셋 간 직접 foreign key처럼 연결하는 구조는 현재 단계에서 기대하기 어렵다.

## 좌표 기반 매핑 가능성

### 판단
- 좌표 기반 연결은 보조 전략으로는 유효하지만, 단독 전략으로 쓰기에는 부족하다.

### 이유
- 버스 정류장은 좌표 기준 매칭이 비교적 현실적이다.
- 지하철은 역사, 출입구, 플랫폼, 운행 정보의 좌표 범위가 다를 수 있어 역명 문맥이 더 중요하다.
- 동일 명칭 다건, 역사 인접 출입구, 방향별 정류장 분리 같은 문제 때문에
  좌표 단독 매칭은 오탐이 날 수 있다.

### 적용 원칙
- 버스:
  - 이름 정규화 후 다건이면 좌표 근접 순으로 후보를 좁힌다.
- 지하철:
  - 좌표보다 노선명 + 역명 + 운행구간 포함 여부를 우선한다.

## 자동 매핑 실패 유형

### 1. 이름 표기 차이
- `서면역`, `서면`, `서면역(2호선)`처럼 표기가 다른 경우
- `역`, `번출구`, 괄호 표기, 중간 구분점 차이

### 2. 동일 명칭 다건
- 같은 정류장명이 상행/하행 각각 존재
- 같은 역명이라도 다른 노선 문맥에서 중복

### 3. 좌표 오차
- 외부 API마다 좌표 기준점이 다를 수 있음
- 역사 중심점과 출입구 좌표가 섞일 수 있음

### 4. 노선/운행 문맥 부족
- 지하철은 역명만 같아도 어느 운행구간 문맥인지 불명확할 수 있음
- 버스는 노선 번호만으로는 동일 정류장 방향 구분이 어려울 수 있음

## 수동 보정이 필요한 대상

- 이름 정규화 후에도 동일 후보가 2건 이상 남는 버스 정류장
- 같은 역명이 다중 노선에 걸쳐 있어 운행구간 문자열만으로 좁혀지지 않는 지하철 역
- ODsay 구간명과 공공데이터 표기 간 차이가 커서 자동 매칭 실패한 케이스
- 출입구 / 역사 / 플랫폼 단위가 혼재되어 앵커를 확정하기 어려운 케이스

MVP에서는 이 케이스를 별도 `manual_mapping_candidate` 목록으로 남기는 방식이 적절하다.

## MVP 적용 전략

### 바로 적용 가능한 것
- `ODsay`를 전체 경로 후보 앵커로 사용
- 버스 구간에 대해 `정류장명 정규화 + BIMS bstopid 확보 + stopArrByBstopid` 연결
- 지하철 구간에 대해 `노선명 + 탑승/하차역명 + 운행구간정거장 포함 여부` 기반 연결
- 지하철 도착/출발 시각은 `TIMETABLE` 성격으로만 표기

### 지금 하지 않는 것
- 세 데이터셋 전체를 완전 자동 통합하는 마스터 테이블 구축
- 역사 내부 출입구/엘리베이터/동선까지 포함한 상세 앵커 자동화
- 실시간 지하철 도착예정처럼 표현하는 기능

## 다음 단계 입력값

Step 4 이후에 바로 넘길 수 있는 입력값은 아래와 같다.

1. `ODsay segment`를 기준으로 `BUS`, `SUBWAY`, `WALK`를 구분하는 규칙
2. 버스 구간용 `정류장명 정규화 규칙`
3. 지하철 구간용 `노선명 + 역명 + 운행구간 포함 여부` 매핑 규칙
4. 자동 매핑 실패 시 남길 `manual_mapping_candidate` 구조

## 결론

이번 단계 기준 연결 전략은 아래처럼 확정하는 것이 적절하다.

- `ODsay`: 경로 후보와 총 이동시간의 기준 앵커
- `부산 BIMS`: 버스 정류장 식별과 실시간 도착시간 보강
- `부산교통공사/odcloud`: 지하철 역명, 정거장 순서, 도착/출발 시각 보강

즉 MVP 연결 전략은
`ODsay를 기준으로 segment를 분해하고, 버스는 BIMS, 지하철은 odcloud로 보강하는 방식`
으로 정리된다.

직접 ID 매핑만으로 해결하려 하지 않고,
`이름 정규화 + 좌표 근접 + 노선 문맥 + 수동 보정 후보 분리`
를 결합한 전략이 현재 단계에서 가장 현실적이다.
