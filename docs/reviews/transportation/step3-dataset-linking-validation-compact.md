# Step 3 Dataset Linking Validation

## 목적
- `부산 BIMS`와 `부산교통공사/odcloud`를 기준 데이터셋으로 두고, `ODsay` 경로 응답의 정류장/역 정보를 실제로 연결할 수 있는지 검증한다.
- 직접 ID 매핑이 불충분한 경우를 가정해 `이름 정규화 + 좌표 근접 + 노선 보조키` 조합 전략이 실제로 동작하는지 확인한다.
- 전략 적용 후에도 매핑되지 않는 지점을 리스트로 남긴다.

## 기준 데이터셋
- 버스 메인: `부산 BIMS`
- 지하철 메인: `부산교통공사/odcloud`
- 경로 후보 입력: `ODsay`

## 1차 검증 재현 명령
```powershell
python C:\Users\SSAFY\poc\scripts\transportation\evaluate_dataset_linking.py
```

## 1차 검증 범위
- 시나리오 1: `부산역 -> 서면`
- 시나리오 2: `서면 -> 해운대`
- 각 시나리오에서 `ODsay searchPubTransPathT` 상위 3개 경로만 사용
- 버스 구간 4개, 지하철 구간 2개를 평가

## 적용한 매핑 규칙
### 버스
1. `ODsay localStationID -> BIMS bstopid` 직접 확인
2. 직접 ID와 무관하게 fallback 매칭 수행
3. fallback 규칙:
   - 정류장명 정규화 비교
   - 좌표 근접도 비교
   - `stopArrByBstopid`의 `lineno` 또는 `lineid`로 노선 보조 검증

### 지하철
1. `odcloud`를 메인 데이터셋으로 사용
2. 직접 ID 매핑 대신 아래 조합으로 연결
   - 노선명 정규화: `부산 1호선 -> 1호선`, `부산 도시철도 1호선 -> 1호선`
   - 출발역명 정규화
   - 도착역명 정규화
   - `운행구간정거장` 문자열 내 포함 여부 확인

## 1차 fallback 판단
### 버스
- `부산역`
  - 정규화 이름 일치
  - 좌표 거리 약 `0.000113`
  - `stopArrByBstopid`에서 노선 `17`, `81` 확인
- `서면역.롯데호텔백화점`
  - 정규화 이름 일치
  - 좌표 거리 약 `0.000111` 또는 `0.000270`
  - `stopArrByBstopid`에서 노선 `17`, `81`, `141`, `31` 확인
- `해운대도시철도역`
  - 정규화 이름 일치
  - 좌표 거리 약 `0.000254`
  - `stopArrByBstopid`에서 노선 `141`, `31` 확인

### 지하철
- `부산역 -> 서면`
  - `부산 1호선`과 `부산 도시철도 1호선`을 같은 라인키로 정규화
  - `운행구간정거장`에 `부산역`, `서면` 모두 포함
- `서면 -> 해운대`
  - `부산 2호선`과 `부산 도시철도 2호선`을 같은 라인키로 정규화
  - `운행구간정거장`에 `서면`, `해운대` 모두 포함

- 없음

### 부산교통공사/odcloud 기준 미매핑
- 없음

## 1차 해석
- 이번 샘플에서는 `ODsay -> BIMS` 버스 정류장 연결이 직접 ID 없이도 fallback 규칙으로 복원 가능했다.
- 지하철은 애초에 공통 ID를 기대하기 어려워 `노선명 정규화 + 역명 포함` 방식이 현실적이며, 이번 샘플에서는 모두 연결됐다.
- 따라서 Step 3 MVP 관점에서는 `BIMS/odcloud를 메인 앵커로 두고, ODsay 응답을 후행 매핑`하는 전략이 유효하다.

### 확장 표본 해석
- 버스는 해운대구 확장 표본 10건에서도 모두 `BIMS` 기준 매핑이 가능했다.

- 따라서 현재 Step 3 기준의 실질적인 잔여 리스크는 `버스 정류장명 변형`보다 `odcloud 기준 데이터의 동해선 커버리지 부재 또는 분리 운영 여부`에 있다.

## 한계
- 확장 검증도 여전히 표본 기반이며, 해운대구 전체 모든 정류장/역 조합을 전수 조사한 것은 아니다.
