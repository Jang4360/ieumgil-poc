# 05_WALK_ROUTE_INPUT_MAPPING_VALIDATION.md

## 단계명

도보 구간을 우리 보행 경로 입력 구조로 넘길 수 있는지 검증

## 목적

대중교통 경로에서 추출한 도보 구간 anchor를 현재 보행 경로 API 입력 구조와 연결할 수 있는지 확인한다.

## 구현 접근

1. Step 4에서 도출한 도보 anchor를 `startPoint`, `endPoint` 형태로 변환해 본다.
2. 현재 보행 경로 API가 받는 좌표 포맷과 좌표계를 기준으로 정규화 규칙을 정리한다.
3. 출입구 좌표, 역사 좌표, 정류장 좌표 중 어떤 값을 우선 사용할지 기준을 검토한다.
4. 스냅핑 또는 좌표 보정이 필요한 케이스를 기록한다.
5. API 연결 시 필요한 최소 입력 구조와 후속 보정 규칙을 나눈다.

## 확인할 것

- 도보 구간 anchor를 `startPoint` / `endPoint` 구조로 만들 수 있는지
- 현재 보행 경로 API 구조와 필드 충돌 없이 연결 가능한지
- 좌표계, 소수점 정밀도, 포맷 정규화가 필요한지
- 출입구 / 정류장 / 역사 좌표 우선순위를 정할 수 있는지
- 보행 네트워크 스냅핑이 필요한 사례가 무엇인지

## 산출물

- 도보 구간 입력 매핑 표
- 좌표 정규화 규칙 메모
- 보행 경로 API 연계 구조 초안
- 보정이 필요한 예외 케이스 목록

## 완료 기준

- 도보 구간을 현재 보행 경로 입력값으로 변환하는 구조 설명 가능
- 최소 좌표 정규화 규칙 1세트 이상 정리
- 연계 불가 또는 추가 보정 필요 케이스 문서화

## 범위 제한

- 실제 API 통합 구현이나 성능 검증은 현재 단계에 포함하지 않는다.
- 출입구 우선순위의 완전한 정책 확정은 하지 않는다.

- 통합 응답에서 재사용할 도보 구간 필드 목록
