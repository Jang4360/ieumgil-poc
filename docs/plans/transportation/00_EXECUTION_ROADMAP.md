# 00_EXECUTION_ROADMAP

## 문서 목적

대중교통 PoC를 어떤 순서와 기준으로 검증할지 통합 기준을 정리한다.

이 문서는 버스, 지하철, 도보 연계를 한 번에 구현하는 문서가 아니라, MVP 도입 가능성을 단계적으로 확인하기 위한 상위 실행 계획 문서다.

---

## 성공 기준

### 최소 성공 기준

- 대중교통 경로 후보를 가져올 수 있다
- 버스 / 지하철 관련 데이터 원천을 식별할 수 있다
- ODsay / BIMS / 지하철 접근성 데이터 간 연결 키 또는 연결 전략을 찾을 수 있다
- 대중교통 경로의 도보 구간 시작점 / 종료점을 추출할 수 있다
- 도보 구간을 우리 보행 경로 입력값으로 넘길 수 있는 구조를 설명할 수 있다

### 확장 성공 기준

- 대중교통 후보 경로를 접근성 기준으로 재정렬할 수 있는 방향성을 찾는다
- 버스 / 지하철 / 도보 구간을 하나의 응답 구조로 묶을 수 있는 방향성을 찾는다
- 저상버스, 엘리베이터, 역사 접근성 같은 요소를 반영할 수 있는 후속 구조를 정의한다

---

## 전체 진행 순서

### Step 1. 대중교통 데이터 원천 후보 조사

무엇을 검증하는가:
- ODsay, BIMS, 지하철 관련 공개 데이터가 실제 PoC 입력 원천이 될 수 있는지 확인한다.

왜 이 순서로 검증하는가:
- 경로 조회, 연결 전략, 도보 연계는 모두 입력 원천의 제공 범위와 식별 키를 알아야만 현실적으로 검토할 수 있다.

어떤 입력 데이터를 확인해야 하는가:
- ODsay의 경로 탐색 응답 범위와 정류장 / 역 / 노선 정보
- BIMS의 버스 노선, 정류장, 운영 정보 제공 범위
- 지하철 역사, 출입구, 엘리베이터, 접근성 관련 공개 데이터
- 각 데이터의 제공 방식, 포맷, 호출 제약, 좌표 포함 여부

어떤 결과가 나오면 성공으로 볼 것인가:
- 버스 / 지하철 후보 경로 조회용 데이터 원천과 접근성 보강용 데이터 원천을 구분해 정리할 수 있다.

어떤 한계가 있으면 후속 검토로 넘길 것인가:
- 실시간성, 요금 정책, 운영기관별 세부 제약은 현재 단계에서 확정하지 않고 후속 검토로 넘긴다.

### Step 2. 버스 / 지하철 경로 후보를 가져올 수 있는지 검증

무엇을 검증하는가:
- 출발지 / 도착지 기준으로 버스, 지하철, 환승 조합이 포함된 대중교통 후보 경로를 가져올 수 있는지 확인한다.

왜 이 순서로 검증하는가:
- 실제 후보 경로를 얻지 못하면 이후의 연결 전략과 도보 연계도 검증할 수 없다.

어떤 입력 데이터를 확인해야 하는가:
- 출발지 / 도착지 좌표 또는 주소 기반 조회 입력값
- 경로 후보 수, 환승 횟수, 소요시간, 정류장 / 역 / 노선 포함 여부
- 응답 내 구간 단위 타입 구분 가능 여부

어떤 결과가 나오면 성공으로 볼 것인가:
- 최소 1개 이상의 대중교통 후보 경로를 조회할 수 있고, 버스 / 지하철 / 환승 관련 핵심 필드를 식별할 수 있다.

어떤 한계가 있으면 후속 검토로 넘길 것인가:
- 요금, 혼잡도, 실시간 도착 정보는 현재 PoC의 필수 성공 조건으로 보지 않는다.

### Step 3. ODsay / BIMS / 지하철 접근성 데이터 연결 전략 검토

무엇을 검증하는가:
- 서로 다른 데이터셋 사이에서 정류장, 역사, 노선 정보를 현실적으로 연결할 수 있는지 검토한다.

왜 이 순서로 검증하는가:
- 후보 경로 응답 구조를 본 뒤에야 어떤 ID와 명칭이 실제로 연결 후보가 되는지 판단할 수 있다.

어떤 입력 데이터를 확인해야 하는가:
- 정류장 ID, 역 ID, 노선명, 정류장명, 역사명, 좌표
- 행정기관 또는 운영기관별 코드 체계 차이
- 동일 명칭 다건 존재 여부와 좌표 오차 범위

어떤 결과가 나오면 성공으로 볼 것인가:
- 완전 일치가 아니더라도 좌표 기반 매칭, 보조 키 매칭, 수동 보정 후보를 조합한 연결 전략을 문서화할 수 있다.

어떤 한계가 있으면 후속 검토로 넘길 것인가:
- 데이터셋 간 ID 불일치, 명칭 표기 차이, 좌표 오차로 인해 100% 자동 매핑이 어려운 지점은 실패 케이스로 기록하고 후속 검토로 넘긴다.

### Step 4. 도보 구간 추출 가능성 검증

무엇을 검증하는가:
- 대중교통 후보 경로에서 출발 도보, 환승 도보, 도착 도보 구간의 시작점과 종료점을 추출할 수 있는지 확인한다.

왜 이 순서로 검증하는가:
- 연결 전략이 어느 정도 정리돼야 도보 anchor가 어떤 정류장 / 역사와 연결되는지 해석할 수 있다.

어떤 입력 데이터를 확인해야 하는가:
- 경로 응답의 세부 구간 타입
- 정류장 / 역사 좌표와 출입구 좌표 존재 여부
- 도보 구간 길이, 시간, 시작 / 종료 지점 표현 방식

어떤 결과가 나오면 성공으로 볼 것인가:
- 대중교통 후보 경로에서 도보 구간별 anchor를 식별하고, 출발 / 환승 / 도착 도보를 구분하는 기준을 정리할 수 있다.

어떤 한계가 있으면 후속 검토로 넘길 것인가:
- 역사 내부 이동, 지하 연결통로, 복합 환승센터 내부 동선처럼 외부 좌표만으로 해석하기 어려운 구간은 후속 검토로 넘긴다.

### Step 5. 도보 구간을 우리 보행 경로 입력 구조로 넘길 수 있는지 검증

무엇을 검증하는가:
- 추출한 도보 구간 anchor를 현재 보행 경로 API 입력 구조로 넘길 수 있는지 확인한다.

왜 이 순서로 검증하는가:
- anchor 추출이 먼저 안정돼야 우리 보행 네트워크와의 연결 조건을 검토할 수 있다.

어떤 입력 데이터를 확인해야 하는가:
- `startPoint`, `endPoint` 형태의 좌표 구조
- 현재 보행 경로 API가 받는 좌표 포맷과 좌표계
- 정류장 / 역사 / 출입구 좌표의 정규화 필요 여부

어떤 결과가 나오면 성공으로 볼 것인가:
- 대중교통 경로의 도보 구간을 우리 보행 경로 입력값으로 변환하는 구조를 설명할 수 있고, 필요한 좌표 정규화 규칙을 정리할 수 있다.

어떤 한계가 있으면 후속 검토로 넘길 것인가:
- snapping 허용 반경, 출입구 우선순위, 지상 / 지하 출발점 보정 규칙은 후속 세부 설계로 넘긴다.

### Step 6. 접근성 정보 반영 가능성 검토

무엇을 검증하는가:
- 저상버스, 역사 엘리베이터, 지하철 접근성, 정류장 / 역 출입구 정보를 현재 단계에서 어디까지 반영할 수 있는지 구분한다.

왜 이 순서로 검증하는가:
- 기본 경로 조회와 도보 연계 가능성이 확인된 뒤에야 접근성 정보를 실제로 어느 구간에 붙일지 판단할 수 있다.

어떤 입력 데이터를 확인해야 하는가:
- 저상버스 운행 여부 관련 데이터
- 역사 엘리베이터 / 에스컬레이터 / 출입구 접근성 정보
- 역 단위와 출입구 단위 데이터의 구분 가능 여부

어떤 결과가 나오면 성공으로 볼 것인가:
- 지금 바로 반영 가능한 값과 후속 확장 값의 경계를 정리하고, 어느 응답 구간에 어떤 접근성 정보를 부착할지 방향을 정할 수 있다.

어떤 한계가 있으면 후속 검토로 넘길 것인가:
- 실시간 저상버스 배차, 역사 내부 세부 동선, 운영기관별 접근성 데이터 품질 차이는 후속 검토 과제로 남긴다.

### Step 7. 통합 응답 구조 방향 검토

무엇을 검증하는가:
- 대중교통 구간과 도보 구간을 하나의 응답 구조로 표현할 수 있는지 검토한다.

왜 이 순서로 검증하는가:
- 데이터 원천, 연결 전략, 도보 연계, 접근성 반영 범위를 확인한 뒤에야 MVP 응답 구조를 현실적으로 정리할 수 있다.

어떤 입력 데이터를 확인해야 하는가:
- 버스 / 지하철 / 도보 각 구간의 필수 필드
- 구간 타입 구분 방식
- 접근성 정보 부착 위치

어떤 결과가 나오면 성공으로 볼 것인가:
- 구간 타입을 `WALK`, `BUS`, `SUBWAY`처럼 분리하고, MVP에서 반드시 내려야 할 필드와 후속 확장 필드를 구분한 통합 응답 방향을 정리할 수 있다.

어떤 한계가 있으면 후속 검토로 넘길 것인가:
- 정교한 점수화, 우선순위 재정렬 로직, 실시간 운영 정보 반영 구조는 MVP 범위 밖으로 남긴다.

### Step 8. 결과 정리

무엇을 검증하는가:
- 전체 검증 결과를 다음 단계 입력값으로 사용할 수 있을 정도로 정리할 수 있는지 확인한다.

왜 이 순서로 검증하는가:
- PoC는 구현보다 판단 기록이 우선이므로, 단계별 성공 / 실패 / 보류 기준을 마지막에 문서로 고정해야 한다.

어떤 입력 데이터를 확인해야 하는가:
- 데이터 원천별 가능 / 불가 항목
- 연결 가능성 및 매핑 전략
- 도보 구간 연계 가능성
- MVP 도입 가능 범위와 후속 과제

어떤 결과가 나오면 성공으로 볼 것인가:
- 대중교통 PoC에서 바로 시도할 범위와 보류할 범위를 문서로 확정할 수 있다.

어떤 한계가 있으면 후속 검토로 넘길 것인가:
- 데이터 부재, 연결 실패, 품질 불일치 사례도 반드시 산출물로 남기고 후속 검토 대상으로 분리한다.

---

## 단계별 구현 원칙

1. 처음부터 모든 대중교통 수단을 완벽히 통합하려 하지 않는다.
2. 데이터 원천 식별 → 후보 경로 조회 → 연결 전략 → 도보 연계 순으로 범위를 좁혀간다.
3. 명칭 완전 일치보다 좌표 기반 연결과 보조 키 전략을 함께 검토한다.
4. 접근성 정보는 지금 바로 반영 가능한 값과 후속 확장 값을 구분한다.
5. 현재 목표는 완성 구현이 아니라 MVP 도입 가능성 검증이다.
6. 실패 케이스와 데이터 불일치 가능성도 산출물로 남긴다.
