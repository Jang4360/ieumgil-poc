# 02_CONSTRAINTS.md

## 기술 제약

1. 현재 PoC 인프라는 아래 3개 컨테이너를 기준으로 한다.
    - backend
    - postgres
    - graphhopper

2. ETL은 현재 단계에서 별도 컨테이너로 운용하지 않는다.
3. 현재 단계에서 Python ETL 구현은 최소화한다.
4. 원천 데이터 검증은 샘플 확인 중심으로 진행한다.

---

## 설계 제약

1. 메인 프로젝트 ERD는 변경하지 않는다.
2. MVP 우선 원칙을 유지한다.
3. 검증되지 않은 확장 구조를 현재 단계의 기본 구조로 채택하지 않는다.

---

## 작업 제약

1. 현재 단계에서는 "부산 OSM 원천 데이터 확보 및 범위 검증" 외의 구현 작업을 확장하지 않는다.
2. 모든 결과는 Markdown 문서로 남긴다.
3. 원천 데이터 출처와 선택 이유는 반드시 기록한다.

### 현재 단계에서 허용하는 준비 작업

현재 단계에서는 원천 데이터 검증을 방해하지 않는 범위에서 아래 준비 작업만 허용한다.

1. `backend`, `postgres`, `graphhopper` 3개 컨테이너 기준의 `docker-compose.yml` 작성
2. `ieumgil-routing-poc`의 최소 실행용 `Dockerfile`, `.dockerignore`, 환경별 설정 파일 작성
3. PostGIS 사용을 위한 최소 초기화 SQL 또는 환경 설정 추가
4. GraphHopper 연동을 위한 최소 `Dockerfile`, 실행 스크립트, 설정 파일 준비
5. 컨테이너 실행에 필요한 `.env` 예시 파일과 실행 명령 문서화
6. 현재 단계 검증 결과와 구현 범위 판단을 `docs/reviews`에 기록

### 현재 단계에서 구현하지 않는 것

아래 항목은 준비 작업으로 보이더라도 현재 단계에서 구현하지 않는다.

1. ETL 전용 컨테이너 추가
2. Python 기반 대량 ETL 스크립트 작성
3. GraphHopper import 자동화 또는 실제 경로 API 검증 로직 추가
4. `road_nodes`, `road_segments` 생성 로직 작성
5. 엔티티, 리포지토리, 서비스 등 비즈니스 코드 확장
6. 메인 프로젝트 ERD를 전제로 한 스키마 변경 또는 마이그레이션 추가
