# 03_EXECUTION_ROADMAP.md

## 문서 목적

Step 1 이후 PoC를 어떤 순서와 방식으로 검증할지 통합 기준을 정리한다.

이 문서는 전체 성공 기준, 단계 순서, 각 단계의 구현 접근을 한 번에 보여주는 상위 계획 문서다.

---

## 성공 기준

### 최소 성공 기준

- 부산 OSM 데이터를 GraphHopper에 적재할 수 있다
- 임의의 출발지/도착지 좌표를 넣으면 보행 경로가 반환된다
- 경로 결과에서 거리, 소요시간, polyline 또는 step을 얻을 수 있다
- 우리 API 응답 형태의 기본 구조로 변환할 수 있다

### 확장 성공 기준

- `SAFE` / `SHORTEST` 두 옵션을 구분해 반환할 수 있다
- 계단 회피 같은 제약을 일부라도 적용할 수 있다
- 위험 구간 정보를 세그먼트 수준으로 붙일 수 있는 방향성을 확인한다

---

## 전체 진행 순서

### Step 1. 부산 OSM 파일 확보

이미 수행한 단계다.

- 부산 OSM 원본 파일 확보
- 사용 범위 메모
- 좌표계 및 포맷 확인

### Step 2. 샘플 지역 먼저 자르기

처음부터 부산 전체를 대상으로 하지 않는다.

예시:
- 서면역 주변 1km ~ 2km

이유:
- 디버깅이 빠름
- 그래프 구조 확인이 쉬움
- DB 적재 검증이 쉬움
- GraphHopper import 시간과 오류 확인이 쉬움

### Step 3. 보행 가능한 OSM 객체만 추출

- 보행 관련 way만 추출
- 교차점 확인
- 계단, 횡단보도, 보행자도로 태그 존재 확인
- 포함/제외 기준표 작성

### Step 4. `road_nodes`, `road_segments` 생성

샘플 구역 기준으로 아래를 검증한다.

- 노드 생성
- 세그먼트 생성
- 길이 계산
- DB 적재

### Step 5. 지도에서 선으로 시각 검증

`road_segments.geom`을 지도에 올려 직접 확인한다.

확인 목적:
- 길이 계산 이상 여부
- 연결 구조 이상 여부
- 세그먼트 길이 과대/과소 여부
- 누락 구간 여부

### Step 6. GraphHopper 샘플 경로 테스트

- 샘플 구역 또는 부산 전체 OSM 기준 GraphHopper import
- 출발지/도착지 좌표 2~3개 테스트
- 경로가 실제 기대와 크게 벗어나지 않는지 확인

### Step 7. 사용자 좌표 스냅핑 검증

- 도로에서 조금 벗어난 GPS 좌표 처리
- 건물 앞, 광장, 도로 근처 좌표 처리
- 보행 불가능 영역 좌표 예외 처리

### Step 8. GraphHopper 응답 분석 및 API 응답 변환

- polyline
- step / instruction
- 거리 / 시간
- snapped point
- `routes[].segments[]` 구조 매핑 가능 여부

### Step 9. `SAFE` / `SHORTEST` 분기 가능성 검토

최초 목표는 아래 수준까지다.

- `SHORTEST`: 기본 보행 경로
- `SAFE`: 계단 회피 또는 일부 회피 룰 적용 가능성 검토

### Step 10. 결과 정리

각 단계 결과는 `docs/reviews/`에 남긴다.

정리 시 반드시 남길 것:
- OSM import 가능 여부
- 기본 보행 경로 성공 여부
- 스냅핑 성공 여부
- API 응답 변환 가능 여부
- `SAFE` / `SHORTEST` 분리 가능 여부
- 위험도 반영이 후처리로 필요한 지점

---

## 단계별 구현 원칙

1. 처음에는 부산 전체보다 샘플 지역으로 먼저 검증한다.
2. ETL과 GraphHopper 검증은 분리해서 확인한다.
3. DB 적재만으로 끝내지 말고 지도로 시각 검증한다.
4. GraphHopper 응답은 원본 그대로 먼저 분석하고, 그 다음 API DTO로 감싼다.
5. `SAFE` / `SHORTEST`는 지금 단계에서 완성 기능이 아니라 구조 분리 가능성 확인이 목표다.
6. 실패 케이스와 예외 처리 방향도 산출물로 남긴다.
