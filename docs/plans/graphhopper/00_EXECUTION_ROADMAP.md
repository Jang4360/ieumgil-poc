# 00_EXECUTION_ROADMAP

## 문서 목적

GraphHopper 기반 보행 경로 PoC를 어떤 순서와 기준으로 검증할지 전체 실행 계획을 정리한다.

이 문서는 단일 단계의 작업 지시서가 아니라, 원천 데이터 확보부터 경로 응답 변환까지의 흐름을 한 번에 확인하기 위한 상위 계획 문서다.

---

## 성공 기준

### 최소 성공 기준

- 부산 OSM 원천 파일을 확보하고 적재 대상을 확정할 수 있다
- 보행 가능 세그먼트 추출 기준을 문서화할 수 있다
- 샘플 구역 기준 `road_nodes`, `road_segments` 생성 가능성을 확인할 수 있다
- GraphHopper에서 기본 보행 경로를 반환받을 수 있다
- GraphHopper 응답을 현재 API 기본 구조로 변환할 수 있다

### 확장 성공 기준

- GPS 오차를 감안한 좌표 스냅핑 기준을 정리할 수 있다
- `SAFE` / `SHORTEST` 분기 가능 방향을 찾을 수 있다
- 위험 구간이나 접근성 속성을 후처리로 결합할 수 있는 구조를 설명할 수 있다

---

## 전체 진행 순서

### Step 1. 부산 OSM 원천 데이터 확보

무엇을 검증하는가:
- 부산 범위 OSM 원천 데이터를 확보하고 PoC 입력 파일로 사용할 수 있는지 확인한다.

왜 이 순서로 검증하는가:
- 이후 모든 단계는 동일한 원천 파일을 기준으로 재현 가능해야 하므로, 시작점부터 입력 데이터를 고정해야 한다.

어떤 입력 데이터를 확인해야 하는가:
- 부산 범위 OSM 파일 후보
- `.osm.pbf` 포맷 여부
- 보행 관련 태그 포함 여부
- 파일 크기와 처리 가능성

어떤 결과가 나오면 성공으로 볼 것인가:
- 적재 대상으로 사용할 부산 OSM 파일 1개 이상을 확정할 수 있다.

어떤 한계가 있으면 후속 검토로 넘길 것인가:
- 최신성 차이, 세부 태그 누락, 운영 반영 주기는 후속 검토로 넘긴다.

### Step 2. 샘플 지역 추출 및 보행 가능 기준 정리

무엇을 검증하는가:
- 부산 전체가 아니라 샘플 구역을 먼저 잘라서 보행 가능한 way 추출 기준을 정리한다.

왜 이 순서로 검증하는가:
- 전체 데이터를 한 번에 다루면 디버깅 비용이 커지므로, 반복 검증 가능한 소규모 샘플이 먼저 필요하다.

어떤 입력 데이터를 확인해야 하는가:
- 샘플 구역 정의
- `highway`, `foot`, `sidewalk`, `access`, `crossing`, `steps`, `elevator` 태그
- 포함 / 제외 / 예외 검토 기준 후보

어떤 결과가 나오면 성공으로 볼 것인가:
- 샘플 구역 1개 이상과 보행 가능 세그먼트 기준표를 정리할 수 있다.

어떤 한계가 있으면 후속 검토로 넘길 것인가:
- 부산 전역 일반화 규칙과 예외 태그의 세부 우선순위는 후속 검토로 넘긴다.

### Step 3. 보행 네트워크 생성 및 DB 적재 검증

무엇을 검증하는가:
- 샘플 구역 데이터를 `road_nodes`, `road_segments` 구조로 변환해 적재할 수 있는지 확인한다.

왜 이 순서로 검증하는가:
- 세그먼트 품질과 연결 구조가 확인돼야 GraphHopper 검증과 후처리 결합 논의를 이어갈 수 있다.

어떤 입력 데이터를 확인해야 하는가:
- 샘플 구역 OSM 데이터
- 보행 가능 세그먼트 추출 규칙
- `road_nodes`, `road_segments` 필수 컬럼 구조

어떤 결과가 나오면 성공으로 볼 것인가:
- 샘플 구역 기준으로 node / segment 생성과 DB 적재 가능 여부를 판단할 수 있다.

어떤 한계가 있으면 후속 검토로 넘길 것인가:
- 대량 적재 성능, 운영 스키마 최적화, 부산 전역 일괄 생성은 후속 검토로 넘긴다.

### Step 4. 세그먼트 시각 품질 검증

무엇을 검증하는가:
- 생성된 `road_segments.geom`을 지도에 올려 연결 구조와 누락 여부를 시각적으로 검증한다.

왜 이 순서로 검증하는가:
- 수치상 적재 성공만으로는 품질을 보장할 수 없으므로, 지도로 직접 확인해야 한다.

어떤 입력 데이터를 확인해야 하는가:
- 샘플 구역 `road_segments.geom`
- 세그먼트 길이와 연결 관계
- 교차점, 막다른 길, 횡단보도 주변 geometry

어떤 결과가 나오면 성공으로 볼 것인가:
- 시각적으로 명확한 오류 여부를 식별하고 규칙 수정 필요 지점을 기록할 수 있다.

어떤 한계가 있으면 후속 검토로 넘길 것인가:
- 지도 UI 고도화나 시각화 자동화는 후속 검토로 넘긴다.

### Step 5. GraphHopper import 및 기본 경로 검증

무엇을 검증하는가:
- GraphHopper에 OSM 데이터를 적재하고 기본 보행 경로를 반환받을 수 있는지 확인한다.

왜 이 순서로 검증하는가:
- 입력 데이터와 세그먼트 기준이 정리된 뒤에야 경로 엔진 자체의 동작을 안정적으로 검증할 수 있다.

어떤 입력 데이터를 확인해야 하는가:
- 부산 또는 샘플 구역 OSM 파일
- 보행용 GraphHopper profile
- 샘플 출발지 / 도착지 좌표쌍

어떤 결과가 나오면 성공으로 볼 것인가:
- 최소 2개 이상의 좌표쌍에서 거리, 시간, geometry가 포함된 보행 경로를 얻을 수 있다.

어떤 한계가 있으면 후속 검토로 넘길 것인가:
- custom model 최적화, 운영 성능 튜닝, 대규모 인프라 설정은 후속 검토로 넘긴다.

### Step 6. 사용자 좌표 스냅핑 검증

무엇을 검증하는가:
- 실제 사용자 GPS 좌표를 보행 가능한 시작점 / 종료점으로 붙일 수 있는지 확인한다.

왜 이 순서로 검증하는가:
- 기본 경로 반환이 가능한 상태여야 스냅핑의 성공 / 실패를 실제 경로 결과와 연결해 판단할 수 있다.

어떤 입력 데이터를 확인해야 하는가:
- 도로에서 약간 벗어난 GPS 좌표 샘플
- snapped point와 원래 좌표 차이
- 실패 처리 기준 거리

어떤 결과가 나오면 성공으로 볼 것인가:
- 일반적인 GPS 오차 범위에서 경로 탐색을 유지할 수 있는 스냅핑 기준을 정리할 수 있다.

어떤 한계가 있으면 후속 검토로 넘길 것인가:
- 고정밀 위치 보정, 실시간 이동 추적, UX 보정 규칙은 후속 검토로 넘긴다.

### Step 7. GraphHopper 응답 분석 및 API 구조 변환

무엇을 검증하는가:
- GraphHopper 원본 응답에서 필요한 필드를 추출해 현재 API 기본 구조로 변환할 수 있는지 확인한다.

왜 이 순서로 검증하는가:
- 경로 계산과 스냅핑이 안정된 뒤에야 응답 매핑 규칙의 현실성을 평가할 수 있다.

어떤 입력 데이터를 확인해야 하는가:
- GraphHopper 원본 응답
- polyline, instruction, distance, time, snapped point 필드
- PoC API DTO 초안

어떤 결과가 나오면 성공으로 볼 것인가:
- 샘플 경로 1건 이상을 `routes[].segments[]` 수준의 구조로 설명할 수 있다.

어떤 한계가 있으면 후속 검토로 넘길 것인가:
- 메인 서비스 최종 API 명세와 인증 / 오류 체계는 후속 검토로 넘긴다.

### Step 8. `SAFE` / `SHORTEST` 분기 가능성 검토

무엇을 검증하는가:
- 기본 최단 경로 외에 안전 우선 경로를 구조적으로 분리할 수 있는지 확인한다.

왜 이 순서로 검증하는가:
- 기본 경로와 응답 변환이 먼저 검증돼야 옵션 분리의 실제 필요 필드를 판단할 수 있다.

어떤 입력 데이터를 확인해야 하는가:
- 기본 보행 경로 결과
- 계단 회피, 일부 구간 회피 같은 제약 후보
- 후처리로 붙일 위험 속성 후보

어떤 결과가 나오면 성공으로 볼 것인가:
- `SAFE` / `SHORTEST`를 분리할 최소 구조와 추가 데이터 필요 지점을 정리할 수 있다.

어떤 한계가 있으면 후속 검토로 넘길 것인가:
- 정교한 안전 점수 모델, 경사도 결합, 공공데이터 완전 통합은 후속 검토로 넘긴다.

### Step 9. 결과 정리

무엇을 검증하는가:
- 전체 검증 결과를 다음 구현 단계의 입력값으로 사용할 수 있을 만큼 정리할 수 있는지 확인한다.

왜 이 순서로 검증하는가:
- PoC는 구현보다 판단 기록이 우선이므로, 성공 / 실패 / 보류 판단을 마지막에 문서로 고정해야 한다.

어떤 입력 데이터를 확인해야 하는가:
- 원천 데이터 선택 이유
- 샘플 구역 및 추출 기준
- GraphHopper 경로 / 스냅핑 / 응답 변환 결과
- 옵션 분리 가능성과 후속 과제

어떤 결과가 나오면 성공으로 볼 것인가:
- 바로 이어갈 구현 범위와 후속 검토 범위를 명확히 구분한 결과 문서를 남길 수 있다.

어떤 한계가 있으면 후속 검토로 넘길 것인가:
- 실패 케이스, 데이터 누락, 품질 불일치도 반드시 후속 검토 항목으로 남긴다.

---

## 단계별 구현 원칙

1. 처음에는 부산 전체보다 샘플 지역으로 먼저 검증한다.
2. 원천 데이터 검증, 네트워크 생성, GraphHopper 검증을 단계별로 분리한다.
3. 수치 결과만 보지 말고 지도 시각 검증을 반드시 포함한다.
4. GraphHopper 응답은 원본 분석을 먼저 하고 그 다음 API 구조로 감싼다.
5. `SAFE` / `SHORTEST`는 완성 기능이 아니라 구조 분리 가능성 검증이 목표다.
6. 실패 케이스와 예외 처리 방향도 산출물로 남긴다.
